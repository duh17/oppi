## Goal

The chat session is the core experience. It must be stutter-free and never freeze in any state — streaming, idle, reconnect, stop, permission gate, compaction, long sessions.

## Audit Results (Feb 9, 2026)

Full code review of: TimelineReducer, DeltaCoalescer, ChatView, ChatItemRow, MarkdownText, ServerConnection, ChatItem.

### Current Strengths
- DeltaCoalescer batches at 33ms — good frame budget
- processBatch() coalesces text/thinking/toolOutput deltas into single upsert
- Streaming assistant uses stable timestamp to avoid Equatable churn
- Thinking uses capped preview buffer — overflow goes to ToolOutputStore
- Code block highlighting is async (`Task.detached`)
- ScrollAnchorState is non-reactive class — avoids feedback loops
- 500-item trim with smart priority (non-conversation first)
- Lazy tool output loading for evicted items
- 5ms perf warning in processBatch

### Identified Risks

#### P0: Freeze Risks
1. **MarkdownText `parseCodeBlocks()` runs on every render** — O(content.length) string scanning on the main thread. For a 10KB assistant message with streaming, this runs every 33ms. Not cached. The `body` property of `MarkdownText` calls `parseCodeBlocks(content)` directly.
   - Fix: Cache parsed blocks, only re-parse when content changes (use `.task(id: content.count)` or equatable wrapper)

2. **`firstIndex(where:)` linear scans in upsert** — `upsertAssistantMessage()` and `upsertThinking()` scan items array to find the current ID. With 500 items, this is O(500) per 33ms flush.
   - Fix: Maintain `itemIndexByID: [String: Int]` lookup table, invalidate on insert/remove

3. **`AttributedString(markdown:)` in prose blocks during streaming** — Apple's markdown parser runs on each prose block. Multiple prose blocks per message, re-parsed every 33ms.
   - Fix: Skip `AttributedString(markdown:)` during streaming, use plain `Text()`. Apply markdown parsing only when `isStreaming == false`

#### P1: Stutter Risks
4. **ForEach re-diff on item mutations** — `ChatItem` is an enum with associated values. When `assistantBuffer` changes, the enum case is reconstructed, ForEach has to diff the value. `LazyVStack` helps but the diff itself costs.
   - Fix: Make ChatItem a reference type or use ID-only ForEach with `@Observable` item objects

5. **`trimIfNeeded()` called in processBatch** — O(n) scan + remove operations. Removing from the middle of an array is O(n).
   - Mostly fine — only triggers when > 500 items. But the sort + remove-by-index-descending pattern is good.

6. **Image decoding in UserMessageBubble** — `attachment.decodedImage` calls `UIImage(data:)` on base64 data inline in the view body.
   - Fix: Pre-decode images when creating the ChatItem, cache UIImage

7. **ImageExtractor.extract() runs on every ToolOutputContent render** — regex scan of potentially large tool output strings.
   - Fix: Cache extraction results, only re-run when output changes

#### P2: Recovery Risks
8. **No gap replay on reconnect** — if WS disconnects and reconnects, events during the gap are lost. `loadBestAvailableHistory` does full reload but may race with live events.
   - Fix: Sequence numbering (TODO-fb28452c covers this)

9. **`reconcileSessionState()` only checks status** — 10s after stop, fetches session from REST but only upserts session metadata. Doesn't reconcile timeline items that may have been added server-side during the gap.
   - Fix: After reconcile, also load trace if session status changed

10. **Ping failure doesn't trigger recovery** — WebSocketClient handles pings internally but if the connection goes dead (no pong), there's no automatic reconnect attempt.
    - Fix: Add connection health watchdog that triggers reconnect

## Implementation Plan

### Phase 1: Zero-stutter streaming (P0 fixes)
- [ ] Cache `parseCodeBlocks()` output — only re-parse when `content.count` changes
- [ ] Skip `AttributedString(markdown:)` during streaming — plain Text for live prose
- [ ] Add `itemIndexByID` lookup in TimelineReducer for O(1) upserts
- [ ] Pre-decode user message images off main thread

### Phase 2: Long-session robustness (P1 fixes)
- [ ] Cache ImageExtractor results in ToolOutputContent
- [ ] Profile ForEach diff cost with 500 items — consider sectioned approach

### Phase 3: Recovery reliability (P2 fixes)
- [ ] WebSocket health watchdog (covered by Phase 1 reliability plan)
- [ ] Gap replay on reconnect (TODO-fb28452c)
- [ ] Reconcile timeline after status change
