## Why
We fixed a lot of send-path flakes, but we still don’t have a strict protocol contract for **idempotent user intents** across reconnect/retry races.

This closes the remaining ghost-send class by making prompt/steer/follow_up delivery explicit and replay-safe.

## Scope
Build on existing `requestId` ACK work in TODO-ec1d9be8 and tighten it to protocol-level invariants.

## Required behavior
For each user intent (`prompt`, `steer`, `follow_up`):

1. iOS sends `clientTurnId` (UUID) + payload
2. Server responds with staged ACKs:
   - `accepted` (persisted + dedupe key recorded)
   - `dispatched` (forwarded to pi stdin)
   - `started` (agent_start observed for that turn)
3. Retries with same `clientTurnId` are idempotent:
   - no duplicate prompt persistence
   - no duplicate dispatch to pi
   - server returns prior ACK state

## Server tasks
- [ ] Add `clientTurnId` to wire types for prompt/steer/follow_up
- [ ] Add per-session dedupe cache keyed by `clientTurnId` (bounded LRU + TTL)
- [ ] Emit staged ACK events (`rpc_result` extension or dedicated `turn_ack`)
- [ ] Guarantee exactly-once persistence semantics for each `clientTurnId`
- [ ] Guarantee at-most-once dispatch to pi for each `clientTurnId`
- [ ] Add tests for duplicate delivery, reconnect resend, and timeout/retry

## iOS tasks
- [ ] Generate and retain `clientTurnId` per send intent
- [ ] Retry with same `clientTurnId` on reconnectable failures
- [ ] UI state machine driven by ACK stage transitions
- [ ] Surface deterministic failure when no ACK progress by timeout budget
- [ ] Add tests for retry/reconnect with same `clientTurnId`

## Acceptance
- [ ] No duplicate user messages/tool runs caused by client retries
- [ ] Every send resolves to one of: delivered (started), explicitly failed, or explicitly canceled
- [ ] No silent send drops in forced reconnect churn tests

## Progress update (2026-02-09, server contract slice)

Implemented the first protocol slice for idempotent turn delivery + staged ACKs:

### Done
- Added `clientTurnId` to wire types for `prompt` / `steer` / `follow_up` (`pi-remote/src/types.ts`).
- Added dedicated staged ACK server event:
  - `turn_ack { command, clientTurnId, stage: accepted|dispatched|started, requestId?, duplicate? }`.
- Added per-session dedupe cache in SessionManager:
  - `TurnDedupeCache` (bounded LRU + TTL) in `pi-remote/src/sessions.ts`.
  - Replays prior ACK stage for duplicate `clientTurnId` retries.
  - Rejects conflicting payload reuse with same `clientTurnId`.
- Prompt persistence moved to SessionManager send path so it is exactly-once per fresh `clientTurnId`.
- Dispatch is at-most-once per `clientTurnId` (duplicate retries do not re-dispatch to pi).
- `started` ACK emitted when `agent_start` is observed (ordered pending-turn queue).
- Server WS handler now passes `clientTurnId/requestId/timestamp` into SessionManager send methods.
- iOS now includes `clientTurnId` on prompt/steer/follow_up sends:
  - `ios/PiRemote/Core/Models/ClientMessage.swift`
  - `ios/PiRemote/Core/Networking/ServerConnection.swift`
