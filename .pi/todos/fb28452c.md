## Context

From Happy architecture review (`ios/HAPPY_COMPARISON.md`). This is our biggest
gap for reconnect reliability. When the iOS app backgrounds and returns, it
cannot recover the tool calls, permission resolutions, or status transitions
that happened while disconnected.

## Problem

Currently `broadcast()` in `sessions.ts` is fire-and-forget — events go to
active WebSocket subscribers and are gone. The `state` event carries a session
snapshot, but:

1. No sequence numbers → client can't detect gaps
2. No event buffer → missed events are unrecoverable
3. No catch-up endpoint → reconnect rebuilds from REST (messages only, no
   tool timeline)
4. `message_end` isn't forwarded to clients → finalized text only available
   via REST refetch

The typical reconnect scenario: phone goes to background for 30 seconds, agent
runs 3 tool calls. When the phone returns, those tool calls are invisible —
the chat timeline has a gap.

---

## Design

### 1. Event Taxonomy: Durable vs Ephemeral

Not all events need sequencing. High-frequency streaming events (deltas) are
ephemeral — they're the body of a message being constructed. Structural events
(a tool started, a message was finalized) are durable — they represent state
transitions that must survive reconnect.

**Durable (sequenced, buffered, recoverable):**

| Event | Why durable |
|-------|------------|
| `agent_start` | Status transition: ready → busy |
| `agent_end` | Status transition: busy → ready |
| `tool_start` | Timeline structure: a tool was invoked |
| `tool_end` | Timeline structure: a tool completed |
| `message_end` | Finalized text checkpoint (NEW — not currently broadcast) |
| `permission_request` | Action required from user |
| `permission_expired` | Action resolved by timeout |
| `permission_cancelled` | Action resolved (tool no longer waiting) |
| `session_ended` | Terminal state |
| `error` | Non-retry errors only |

**Ephemeral (unsequenced, not buffered):**

| Event | Why ephemeral |
|-------|--------------|
| `text_delta` | Token-by-token text (reconstructed from `message_end`) |
| `thinking_delta` | Token-by-token thinking (transient by nature) |
| `tool_output` | Streaming tool output (available from session trace) |
| `state` | Snapshot, not a state change — serves a different purpose |
| `connected` | Connection-specific, not a session event |
| `extension_ui_notification` | Transient status display |

After reconnect, the client rebuilds the timeline from durable events. It won't
see the streaming deltas that happened while offline, but it WILL see: which
tools ran, what the finalized messages said, and what permissions were
requested/resolved. That's the important structural information.

### 2. Wire Format

Add `seq` as an **optional field** on the existing `ServerMessage` union.
Durable events include it; ephemeral events omit it. This is backward-
compatible — old clients ignore the field.

```typescript
// types.ts — add seq to durable event types

export type ServerMessage =
  // Durable (sequenced)
  | { type: "agent_start"; seq: number }
  | { type: "agent_end"; seq: number }
  | { type: "tool_start"; tool: string; args: Record<string, unknown>; seq: number }
  | { type: "tool_end"; tool: string; seq: number }
  | { type: "message_end"; role: "user" | "assistant"; content: string; seq: number }
  | { type: "permission_request"; seq: number; id: string; sessionId: string; /* ... */ }
  | { type: "permission_expired"; seq: number; id: string; reason: string }
  | { type: "permission_cancelled"; seq: number; id: string }
  | { type: "session_ended"; seq: number; reason: string }
  | { type: "error"; seq: number; error: string }

  // Ephemeral (no seq)
  | { type: "connected"; session: Session; currentSeq: number }
  | { type: "state"; session: Session }
  | { type: "text_delta"; delta: string }
  | { type: "thinking_delta"; delta: string }
  | { type: "tool_output"; output: string; isError?: boolean }
  | { type: "extension_ui_request"; /* ... */ }
  | { type: "extension_ui_notification"; /* ... */ }
```

Note: `connected` gains a `currentSeq` field so the client knows the server's
current position immediately on connect, before deciding whether catch-up is
needed.

### 3. Seq Counter & Event Ring (SessionManager)

Add to `ActiveSession`:

```typescript
interface ActiveSession {
  // ... existing fields ...

  /** Per-session monotonic sequence counter */
  seq: number;

  /** Ring buffer of recent durable events for catch-up */
  eventRing: EventRing;
}
```

The `EventRing` is a bounded in-memory buffer:

```typescript
interface SequencedEvent {
  seq: number;
  event: ServerMessage;  // The full event payload
  timestamp: number;
}

class EventRing {
  private events: SequencedEvent[] = [];
  private readonly capacity: number;

  constructor(capacity = 500) {
    this.capacity = capacity;
  }

  push(event: SequencedEvent): void {
    this.events.push(event);
    if (this.events.length > this.capacity) {
      this.events.shift();
    }
  }

  /** Return events with seq > sinceSeq */
  since(sinceSeq: number): SequencedEvent[] {
    // Binary search would be faster, but 500 items is fine with findIndex
    const idx = this.events.findIndex(e => e.seq > sinceSeq);
    return idx === -1 ? [] : this.events.slice(idx);
  }

  get currentSeq(): number {
    const last = this.events[this.events.length - 1];
    return last?.seq ?? 0;
  }

  get oldestSeq(): number {
    return this.events[0]?.seq ?? 0;
  }

  /** Can we serve a catch-up from sinceSeq? */
  canServe(sinceSeq: number): boolean {
    // Can serve if sinceSeq >= oldest-1 (i.e., no gap)
    return this.events.length === 0 || sinceSeq >= this.oldestSeq - 1;
  }
}
```

**Ring size of 500:** A typical agent turn produces ~10-20 durable events
(agent_start + N×tool_start/tool_end + message_end + agent_end). 500 events
covers ~25-50 agent turns — more than enough for typical background durations.

### 4. Broadcast Changes (SessionManager)

Split broadcast into two paths:

```typescript
/** Broadcast a durable event — sequenced and buffered */
private broadcastDurable(key: string, event: ServerMessage): void {
  const active = this.active.get(key);
  if (!active) return;

  active.seq += 1;
  const sequenced = { ...event, seq: active.seq };

  active.eventRing.push({
    seq: active.seq,
    event: sequenced,
    timestamp: Date.now(),
  });

  for (const cb of active.subscribers) {
    try { cb(sequenced); } catch (err) { console.error("Subscriber error:", err); }
  }
}

/** Broadcast an ephemeral event — no seq, no buffer */
private broadcastEphemeral(key: string, event: ServerMessage): void {
  const active = this.active.get(key);
  if (!active) return;

  for (const cb of active.subscribers) {
    try { cb(event); } catch (err) { console.error("Subscriber error:", err); }
  }
}
```

The existing `broadcast()` call sites change based on event type. In
`translateEvent()` and `handleRpcLine()`, route through the correct path.

### 5. New `message_end` Broadcast

Currently `message_end` from pi RPC is handled in `updateSessionFromEvent()`
(persists assistant text to storage, updates stats) but is NOT broadcast as
its own event — only a `state` snapshot follows. Add a durable broadcast:

```typescript
case "message_end": {
  const assistantText = this.extractAssistantText(event.message);
  const role = event.message?.role;

  if (role === "assistant" && assistantText) {
    this.broadcastDurable(key, {
      type: "message_end",
      role: "assistant",
      content: assistantText,
    });
  }
  break;
}
```

This gives the iOS client the finalized message text on reconnect catch-up,
without needing to replay the delta stream.

### 6. Catch-Up REST Endpoint

```
GET /sessions/:id/events?since=<seq>
Authorization: Bearer <token>

Response 200:
{
  "events": [
    { "seq": 43, "type": "tool_start", "tool": "bash", ... },
    { "seq": 44, "type": "tool_end", "tool": "bash", ... },
    ...
  ],
  "currentSeq": 47,
  "session": { ... },       // Current session snapshot
  "catchUpComplete": true   // false if ring buffer couldn't serve full history
}
```

When `catchUpComplete` is false (requested seq fell off the ring buffer), the
client must do a full rebuild: `GET /sessions/:id` for messages + trace for
tool history. This is the cold path — rare, only after long disconnections or
server restarts.

Implementation in `server.ts`:

```typescript
// GET /sessions/:id/events?since=<seq>
const eventsMatch = path.match(/^\/sessions\/([^/]+)\/events$/);
if (eventsMatch && method === "GET") {
  const sessionId = eventsMatch[1];
  const sinceParam = url.searchParams.get("since");
  const sinceSeq = sinceParam ? parseInt(sinceParam, 10) : 0;

  // Need SessionManager access for the event ring
  const catchUp = this.sessions.getCatchUp(user.id, sessionId, sinceSeq);
  if (!catchUp) {
    this.error(res, 404, "Session not found or not active");
    return;
  }

  this.json(res, catchUp);
  return;
}
```

SessionManager method:

```typescript
getCatchUp(userId: string, sessionId: string, sinceSeq: number): CatchUpResponse | null {
  const key = `${userId}/${sessionId}`;
  const active = this.active.get(key);
  if (!active) return null;

  const canServe = active.eventRing.canServe(sinceSeq);
  const events = canServe ? active.eventRing.since(sinceSeq) : [];

  return {
    events: events.map(e => e.event),
    currentSeq: active.seq,
    session: active.session,
    catchUpComplete: canServe,
  };
}
```

### 7. Enhanced `connected` Message

When the WebSocket connects, include `currentSeq` so the client can decide
whether catch-up is needed without a separate REST call:

```typescript
send({
  type: "connected",
  session: activeSession,
  currentSeq: active.seq,  // NEW
});
```

Client logic on connect:
- If `currentSeq === lastSeenSeq` → fully in sync, no catch-up needed
- If `currentSeq > lastSeenSeq` → call `GET /events?since=lastSeenSeq`
- If `lastSeenSeq === 0` (first connect) → no catch-up, start fresh

### 8. Reconnect Flow (iOS Client)

```
Phone foregrounded / WebSocket reconnects:

1. Connect WebSocket → receive { type: "connected", currentSeq: 47 }

2. Compare with stored lastSeenSeq (e.g., 42):
   - If equal → done, resume streaming
   - If different → continue to step 3

3. GET /sessions/:id/events?since=42
   → { events: [seq 43..47], catchUpComplete: true }

4. Process catch-up events through TimelineReducer
   (same code path as live events)

5. Update lastSeenSeq = 47

6. Resume processing live WebSocket events
   (dedup: skip any with seq <= lastSeenSeq)

Edge case — catchUpComplete: false:
   → Fall back to GET /sessions/:id for full message rebuild
   → GET /sessions/:id/trace for tool history
   → Rebuild timeline from scratch
   → Set lastSeenSeq = currentSeq from connected message
```

### 9. Seq Deduplication Window

Between the REST catch-up response and WebSocket resume, there's a small
overlap window where the same event could arrive via both paths. The client
deduplicates by tracking `lastSeenSeq`:

```swift
func processEvent(_ event: ServerMessage) {
    // Skip events already seen via catch-up
    if let seq = event.seq, seq <= lastSeenSeq {
        return
    }

    if let seq = event.seq {
        lastSeenSeq = seq
    }

    // Normal processing...
    reducer.process(event)
}
```

### 10. Seq Persistence

**Server side:** The seq counter lives in-memory on `ActiveSession`. It resets
to 0 when the session process exits and restarts. This is fine because:
- The event ring is also in-memory (both reset together)
- After server restart, clients get `currentSeq: 0` on connect
- Client sees `currentSeq < lastSeenSeq` → full rebuild (cold path)

We do NOT persist seq to disk in v1. The complexity isn't worth it — server
restarts are rare, and the cold path (full rebuild from REST) works.

**Client side:** `lastSeenSeq` persisted per session in UserDefaults (not
Keychain — it's not sensitive). Lost on app reinstall, which is fine (triggers
full rebuild).

## What This Does NOT Cover (Future Work)

- **On-disk event log**: For catch-up after server restart. v2 enhancement.
  Could use JSONL append log alongside the session trace files.
- **Cross-session seq**: User-level sequencing for "permission on session B
  while viewing session A". Not needed until multi-session notification feed.
- **Optimistic concurrency**: Version fields on session metadata for conflict
  detection. Separate concern from event sequencing.
- **Ephemeral event recovery**: Thinking/delta streams are intentionally not
  recoverable. After reconnect, the finalized text (from `message_end`) is
  sufficient.

## Implementation Order

1. `EventRing` class (standalone, easy to unit test)
2. `message_end` broadcast in `sessions.ts`
3. `seq` counter + durable/ephemeral split in `sessions.ts`
4. `currentSeq` in `connected` message
5. `GET /events?since=` endpoint in `server.ts`
6. Type updates in `types.ts`
7. iOS: `ServerMessage` decode changes
8. iOS: `lastSeenSeq` tracking + reconnect catch-up flow
9. iOS: `TimelineReducer` handles `message_end`

Steps 1-6 are server-only and can ship independently. Steps 7-9 are iOS and
depend on server changes.
