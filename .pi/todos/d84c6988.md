## Problem

E2E test shows 0 permission requests — the permission-gate extension inside the container never connects to the gate Unix socket. Tools execute without going through the approval flow.

## Symptoms

- `test-e2e.ts` Run 2 (`npm --version`, which should trigger "ask") completes with no `permission_request` WS events
- Server logs show gate socket created but no `guard_ready` event
- Pi exits normally — extension fails silently

## Likely Causes (investigate in order)

### 1. Extension not installed/loaded in container
- `SandboxManager.initUser()` copies extension to `<piDir>/extensions/permission-gate/`
- But pi may need a `settings.json` with the extension registered to load it
- Check: does pi auto-discover extensions in `~/.pi/extensions/`? Or does it need explicit config?
- The host's `settings.json` is a symlink to dotfiles — NOT synced into the sandbox

### 2. Socket path mismatch
- Server creates socket at `<piDir>/gate/<sessionId>.sock` on host
- Container env var `PI_REMOTE_GATE_SOCK` is set to `/home/pi/.pi/gate/<sessionId>.sock`
- After the mount-level fix, `<piDir>/` is mounted as `/home/pi/.pi/`, so `gate/` should be visible
- Verify: does the socket file actually appear inside the container at the expected path?

### 3. Socket timing
- Gate socket is created by `sessions.ts` during `spawnPi()`, before container boots
- Extension connects on `before_agent_start` — should happen after boot
- But: does the socket survive the bind mount? Apple containers may snapshot the dir at mount time

### 4. Extension API mismatch
- Extension uses `pi.on("tool_call", ...)` and `pi.on("before_agent_start", ...)`
- These hook names may have changed in pi v0.52.7 (version in container)
- Check pi extension docs for current hook names
