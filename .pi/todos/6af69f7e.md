## Done

### Changes

**`pi-remote/src/types.ts`** — added `warnings?: string[]` to `Session` interface

**`pi-remote/src/sandbox.ts`** — added `validateSession()` method:
- Checks permission gate extension dir, index.ts, package.json (errors = hard fail)
- Checks memory.ts when `memoryEnabled=true` (warning = degraded)
- Checks auth.json (warning)

**`pi-remote/src/sessions.ts`** — two health check mechanisms:
1. **Pre-spawn validation**: after `spawnPi()`, calls `validateSession()`. Errors → session set to "error" status, container stopped, throws. Warnings → stored on session, logged.
2. **Post-first-prompt guard check**: 10s after first prompt, checks gate state. If extension hasn't connected ("unguarded"), adds warning and broadcasts state update. Extension connects in `before_agent_start` so we can't check earlier.

**`pi-remote/test-bootstrap.ts`** — 24 assertions across 12 test cases:
- Healthy session (with/without memory)
- Missing gate dir / index.ts / package.json / both
- Memory enabled but missing (warning) vs disabled (no warning)
- Missing auth (warning)
- Combined errors + warnings
- Empty sandbox dir

### Decisions
- Permission gate missing → **hard error** (session won't start). Without it every tool call blocks.
- Memory extension missing → **warning** (session starts, no memory). Non-critical.
- Auth missing → **warning** (session starts, API calls may fail).

cd pi-remote && npx tsx test-bootstrap.ts
