## Why
Current stop UX is still partly heuristic. Users can get “is it actually stopping?” uncertainty and may force-kill.

We need a protocol-level lifecycle contract so control actions are deterministic, not vibes-based.

## Lifecycle states
Session runtime state must be explicit and server-authoritative:
- `ready`
- `running`
- `stopping`
- `error`

## Control events
Add explicit server messages for control transitions:
- `stop_requested` (server accepted stop command)
- `stop_confirmed` (agent_end or forced termination complete)
- `stop_failed` (stop attempt failed)

Optional: include `reason` + `source` (`user`, `timeout`, `policy`, `server`).

## Server tasks
- [ ] Add lifecycle/control event types in `types.ts`
- [ ] Move stop handling to two-phase flow: request -> confirm/fail
- [ ] Ensure `state` snapshot and event stream agree on lifecycle
- [ ] Add timeout escalation path: if graceful stop exceeds budget, force terminate and emit final outcome
- [ ] Add tests: duplicate stop taps, stop during reconnect, stop during tool loop

## iOS tasks
- [ ] Render stop progress from explicit lifecycle state (`stopping`)
- [ ] Disable duplicate stop actions while `stopping`
- [ ] Show terminal outcome from `stop_confirmed`/`stop_failed`
- [ ] Add tests for stop UX determinism under reconnect churn

## Acceptance
- [ ] After tapping stop, UI immediately shows `stopping`
- [ ] User always receives explicit terminal outcome (`stop_confirmed` or `stop_failed`)
- [ ] No ambiguous busy state after stop path finishes
