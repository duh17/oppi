## Context

Core runtime refactor from IMPLEMENTATION.md Phase 0. Container lifecycle
moves from session-scoped to workspace-scoped.

Tracker: TODO-748506dc
Architecture: WORKSPACE-CONTAINERS.md
Acceptance criteria: IMPLEMENTATION.md § Phase 0

## Deliverables

1. **Workspace runtime owner** — new `workspace-runtime.ts` or refactor of
   `sessions.ts` to manage `ActiveWorkspace` → `ActiveSession[]`.

2. **Container lifecycle keyed by workspaceId** — sandbox layout becomes
   `~/.pi-remote/sandboxes/<userId>/<workspaceId>/` instead of `.../<sessionId>/`.

3. **Multi-session concurrency** — multiple pi processes per workspace, each with:
   - Own gate port
   - Own RPC correlation maps
   - Shared workspace filesystem + agent home

4. **Lifecycle locks:**
   - Per-workspace mutex for start/stop/delete
   - Per-session mutex for spawn/stop/resume
   - Reject spawn when workspace is stopping

5. **Configurable limits:**
   - maxSessionsPerWorkspace (default: 3)
   - maxSessionsGlobal (default: 5)
   - sessionIdleTimeoutMs (default: 600000)
   - workspaceIdleTimeoutMs (default: 1800000)

6. **Data migration** — detect legacy session-scoped sandboxes on startup,
   migrate to workspace-scoped layout. Write migration marker. Keep rollback
   metadata.

## Acceptance Criteria (from IMPLEMENTATION.md)

- Two sessions in same workspace can run concurrently
- They see the same workspace files and skills
- Stopping one session does not stop sibling sessions
- Stopping workspace cleanly stops all sessions then container
- Gate ports are released on all exit/error paths
- Existing single-session behavior remains functional
