## Context

IMPLEMENTATION.md Phase 3. Fork creates a new session from a specific point
in an existing conversation. New pi process, original keeps running.

Tracker: TODO-748506dc
Depends on: Phase 1 (API migration)

## Deliverables

### Server
1. **REST endpoint:** `POST /workspaces/:wid/sessions/:sid/fork`
   - Accepts `{ entryId: string }`
   - Validates entryId against source session JSONL
   - Copies ancestor chain to new JSONL file
   - Spawns new pi process from fork file
   - Returns `{ sessionId, prefillText }`

2. **Fork lineage metadata** — persist source session + entryId on forked session.

### iOS
3. **Fork action** — long-press user message → "Fork from here"
4. **Fork UX** — opens new session with pre-filled composer (original user message)
5. **Lineage display** — forked session shows "forked from Session A"

## Acceptance Criteria (from IMPLEMENTATION.md)

- Forked session starts without disrupting source session
- Fork lineage displayed in session metadata/UI
- Invalid entryId fails with deterministic API error
