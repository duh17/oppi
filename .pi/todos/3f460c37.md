## Problem

Our `parseJsonl` in `trace.ts` only handles `type: "message"` entries and does a linear scan. Pi TUI uses `buildSessionContext()` which walks a parentId tree from leaf to root and handles compaction.

**Real impact:** A session with 3 compactions and 1074 messages ‚Äî pi TUI shows 119 items, our iOS app shows all 1074. That's 9x the content, including messages the LLM no longer has in context.

## What pi TUI does (`buildSessionContext`)

1. Parse ALL entry types: `session`, `message`, `compaction`, `model_change`, `thinking_level_change`, `branch_summary`, `custom_message`, `label`, `session_info`
2. Build id ‚Üí entry index
3. Walk parentId chain from leaf to root
4. Handle compaction: summary message + kept messages (from `firstKeptEntryId`) + post-compaction messages
5. Skip pre-compaction messages entirely

## What we do (`parseJsonl`)

1. Only parse `type: \"message\"` entries
2. Linear scan ‚Äî no tree walk
3. No compaction awareness ‚Äî shows ALL messages
4. Missing entry types: `compaction`, `model_change`, `thinking_level_change`, `branch_summary`, `custom_message`

## Fix

### Server (`trace.ts`):
- Rewrite `parseJsonl` ‚Üí `buildSessionContext` to match pi's algorithm
- Parse all entry types, build parentId index, walk leaf‚Üíroot
- Handle compaction: emit summary + kept + post-compaction only
- Emit `model_change` and `thinking_level_change` as metadata
- Rename exported function for clarity

### iOS:
- Rename `loadFromTrace` ‚Üí `loadSession` (match pi CLI terminology)
- Handle any new trace event types from the enriched context

### Terminology alignment:
- \"trace\" ‚Üí \"session context\" for the resolved view
- Keep \"trace\" for the raw JSONL concept
