## Goal

Migrate ad-hoc test scripts to vitest, then decompose oversized functions with test coverage as safety net.

## Part 1: Vitest migration

### Setup
- [ ] `npm i -D vitest` in pi-remote
- [ ] Add `"test": "vitest run"` and `"test:watch": "vitest"` to package.json scripts
- [ ] Create `vitest.config.ts` (resolve .js → .ts imports, timeout 10s)

### Migrate existing tests

Each migration: rewrite hand-rolled assert/test into `describe`/`it`/`expect`, keep test logic intact.

| Source | Target | Tests | Notes |
|--------|--------|-------|-------|
| `test-policy.ts` (240L) | `tests/policy.test.ts` | 25 tests: bash parsing (6), container preset (12), hard denies (5), restricted (3), display (2) | Pure unit tests, no I/O. Easiest migration. |
| `test-sandbox-sync.ts` (395L) | `tests/sync.test.ts` | 16 tests: copyFileDereferenced (4), syncFile (4), syncOptionalFile (4), isNewer (1), regressions (3) | Uses tmp dirs + real fs. Map `passed/failed` counters to vitest assertions. |
| `test-bootstrap.ts` (305L) | `tests/bootstrap.test.ts` | 12 tests: healthy session variants (3), missing gate (4), memory (3), auth (1), combined (1) | Instantiates real `SandboxManager` with custom `sandboxBaseDir`. Uses tmp dirs. |
| `test-gate.ts` (137L) | `tests/gate.test.ts` | 5 tests: guard handshake, auto-allow, hard-deny, ask→approve, heartbeat | Integration: creates real `GateServer` + TCP `net.createConnection`. Needs `afterEach` cleanup. Gate uses `createSessionSocket` which binds TCP port. |

**Do NOT migrate:**
- `test-e2e.ts` (932L) — needs running server + containers + LLM API. Keep as standalone `npx tsx test-e2e.ts`.
- `test-gate-client.ts` (173L) — interactive CLI tool, not a test suite.

### New tests to add

- [ ] `tests/types.test.ts` — conformance: verify unknown message types don't crash JSON parse/dispatch

## Part 2: Decompose large functions

### Priority targets

**A. `sessions.ts:startSession` (L103-361, 258 lines)**

Currently does: lookup session, call `spawnPi`, validate sandbox, handle errors, create ActiveSession, set status, persist, start idle timer.

Extract:
- `spawnPi()` already exists as separate private method (good)
- The 258 lines includes spawnPi inline in the line count analysis. Actual `startSession` body is ~60 lines (L103-162). **Not as bad as the awk analysis suggested** — awk counted through nested methods.

**Revised assessment:** `startSession` itself is ~60 lines — reasonable. The real unit is `spawnPi` at ~100 lines. `spawnPi` mixes container args, readline wiring, and readiness probing. Consider extracting:
- `buildContainerArgs(session, gatePort, workspace)` → string[] — pure, testable
- Keep readline/probe wiring in `spawnPi` (it's inherently sequential)

**B. `sandbox.ts:initSession` (L220-390, ~170 lines)**

Currently does: create dirs, sync auth/models/settings, install extensions, sync skills, create shims, write fetch allowlist, generate system prompt.

Each step is already a private method call. The method reads like a checklist — **it's a coordinator, not a monolith**. Decomposing further would just scatter the checklist across indirections.

**Revised assessment:** No decomposition needed. The individual private methods (`syncSkills`, `createSkillShims`, `writeFetchAllowlist`, `generateSystemPrompt`) are already extracted. `initSession` is readable as-is.

**C. `sandbox.ts:generateSystemPrompt` (L668-773, ~105 lines)**

This is a big template string builder. Options:
1. Move template to a separate `.md` template file with mustache-like placeholders → overkill for now
2. Keep inline but extract data-gathering into a helper that returns a typed context object → cleaner

**Revised assessment:** Template logic is inherently verbose. Leave as-is. The only concern is it's not independently testable, but the output is a string we could snapshot-test.

**D. `server.ts:handleHttp` (L301-540, ~240 lines)**

This is one big `if/else` chain routing HTTP requests. Classic pattern for raw `http.createServer`.

Extract route handlers into methods:
```typescript
// Current: inline if/else chain
if (path === "/sessions" && method === "POST") { ... 30 lines ... }

// Proposed: method per route
private async handleCreateSession(user, req, res) { ... }
```

This is the highest-value decomposition — each route becomes independently readable and testable.

**Concrete extraction plan:**
- `handleGetMe(user, res)`
- `handleListModels(res)`
- `handleListSkills(res)` / `handleRescanSkills(res)`
- `handleListWorkspaces(user, res)` / `handleCreateWorkspace(user, req, res)` / `handleGetWorkspace(...)` / `handleUpdateWorkspace(...)` / `handleDeleteWorkspace(...)`
- `handleRegisterDeviceToken(user, req, res)` / `handleDeleteDeviceToken(user, req, res)`
- `handleListSessions(user, res)` / `handleCreateSession(user, req, res)`
- `handleStopSession(user, sessionId, res)`
- `handleGetSessionTrace(user, sessionId, res)`
- `handleGetSession(user, sessionId, res)` / `handleDeleteSession(user, sessionId, res)`

Then `handleHttp` becomes a thin router (~40 lines of pattern matching → method dispatch).

## Key Insight

The awk analysis overstated the problem. The codebase is better-structured than raw line counts suggest:
- `sessions.ts` already has good method extraction (spawnPi, handleRpcLine, translateEvent, etc.)
- `sandbox.ts:initSession` is a coordinator calling well-named private methods
- `server.ts:handleHttp` is the real decomposition target — 240-line if/else chain
- Template strings (system prompt) are verbose but not complex

The biggest wins are: (1) vitest migration for confidence, (2) server.ts route extraction for readability.
